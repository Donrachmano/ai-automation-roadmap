AGENTIC WORKFLOWS

Dari LLM call biasa menjadi agent yang merencanakan, memakai tool, dan mengingat.

Step 01 / Fondasi

LLM Biasa vs Agent

Memahami batas antara "memanggil model" dan "menjalankan agent"

Di Sesi 1 kamu sudah bisa memanggil Claude/OpenAI API: kirim prompt, terima jawaban. Itu satu langkah, satu output. Agent berbeda: ia memutuskan langkah berikutnya sendiri berdasarkan hasil sebelumnya, sampai tujuan tercapai.

Perbedaan inti

  • LLM call: input -> output. Tidak ada aksi ke dunia luar.
  • Agent: loop think -> act -> observe -> repeat sampai selesai.
  • Agent bisa memanggil tool (cari web, baca DB, hitung) lalu memakai hasilnya untuk berpikir lagi.
  • Agent punya memory supaya tahu apa yang sudah dikerjakan.
Mental model

Bayangkan LLM = karyawan yang menjawab satu pertanyaan. Agent = karyawan yang kamu beri tujuan ("buat laporan riset topik X"), lalu ia sendiri yang menentukan: cari dulu, baca, rangkum, tulis. Tugasmu sebagai automation engineer: merancang loop dan tool-nya, bukan mengerjakan tiap langkah manual.

Anatomi sebuah agent

agent-loop.jsjs
// Loop agent paling dasar (pseudocode yang bisa dijalankan)
async function runAgent(goal) {
  let state = { goal, steps: [], done: false };
  while (!state.done && state.steps.length < 8) {
    const decision = await think(state);   // LLM memilih aksi
    const result   = await act(decision);   // jalankan tool
    state.steps.push({ decision, result }); // observe
    state.done = decision.type === "finish";
  }
  return state;
}
Konteks n8n

Di n8n, loop ini tidak kamu tulis manual. Node AI Agent sudah membungkus loop think-act-observe. Tugasmu: pasang Chat Model, daftarkan Tools, dan colok Memory. Sesi ini mengajarkan apa yang terjadi di dalam node itu supaya kamu bisa debug saat agent "nyasar".

Step 02 / Pattern

AI Agent Design Patterns

Pola berpikir yang membuat agent andal, bukan asal jalan

Agent yang baik bukan satu prompt panjang. Ada pola desain yang sudah terbukti. Empat yang paling sering dipakai di automation:

1. ReAct (Reason + Act)

Agent menuliskan alasan sebelum tiap aksi: "Aku perlu data harga, jadi aku panggil tool weather". Membuat keputusan transparan dan mudah di-debug.

2. Planning (rencana dulu, eksekusi kemudian)

Agent membuat daftar langkah di awal, lalu mengeksekusinya satu per satu. Cocok untuk tugas panjang seperti research report.

3. Reflection (kritik diri)

Setelah menghasilkan output, agent mengevaluasi kualitasnya sendiri dan memperbaiki. "Laporan ini kurang sumber, tambahkan 2 referensi."

4. Routing (pilih jalur)

Agent mengklasifikasi input lalu mengarahkan ke handler yang tepat. Mirip switchboard.

// Bentuk output ReAct yang diminta dari LLM
const step = {

Step 03 / Tools

Tool Calling & Function Calling

Cara agent menyentuh dunia nyata

Tool calling = kamu mendeskripsikan fungsi ke LLM (nama, kegunaan, parameter). LLM tidak menjalankan fungsi; ia memilih fungsi mana dan mengisi argumennya. Kode kamu yang mengeksekusi, lalu hasilnya dikembalikan ke LLM.

Alur tool calling

  • 1. Kamu kirim prompt + daftar tool (schema).
  • 2. LLM balas: "panggil get_weather dengan {city: 'Jakarta'}".
  • 3. Kode kamu jalankan fungsi sungguhan, dapat hasil.
  • 4. Hasil dikirim balik ke LLM untuk jawaban final.
tool-schema.jsjs
// Schema tool yang dikirim ke Claude/OpenAI
const tools = [{
  name: "get_weather",
  description: "Ambil cuaca terkini sebuah kota",
  input_schema: {
    type: "object",
    properties: {
      city: { type: "string", description: "Nama kota" }
    },
    required: ["city"]
  }
}];

Step 04 / Orchestration

Multi-Agent Orchestration

Beberapa agent kecil mengalahkan satu agent raksasa

Daripada satu agent mengerjakan semuanya, pecah jadi beberapa agent spesialis dengan satu agent orchestrator yang membagi tugas. Lebih mudah di-debug, di-test, dan tiap agent punya prompt yang fokus.

Pola umum

  • Orchestrator -> Worker: satu agent membagi tugas ke beberapa worker, lalu menggabungkan.
  • Pipeline: output agent A jadi input agent B (riset -> tulis -> edit).
  • Supervisor: agent peninjau memeriksa hasil sebelum keluar.
pipeline.jsjs
// Pipeline 3 agent: riset -> tulis -> edit
async function pipeline(topic) {
  const facts  = await researchAgent(topic);
  const draft  = await writerAgent(facts);
  const final  = await editorAgent(draft);
  return final;
}

Step 05 / Kontrol

Human-in-the-Loop Design

Kapan agent harus berhenti dan menunggu manusia

Agent otonom itu bagus, sampai ia mengirim email salah ke 500 klien. Human-in-the-loop (HITL) menaruh titik persetujuan manusia di langkah berisiko: kirim uang, hapus data, balas pelanggan.

Kapan butuh HITL

  • Aksi irreversible (hapus, kirim, bayar).
  • Output menyentuh orang luar (email klien, posting publik).
  • Agent tidak yakin (confidence rendah) - minta konfirmasi.
  • Nilai transaksi melewati threshold tertentu.

Step 06 / Memory

Agent Memory: Short & Long Term

Agar agent tidak pikun di tengah percakapan

Tanpa memory, tiap langkah agent seperti amnesia. Dua jenis memory:

Short-term (working memory)

Riwayat percakapan/langkah dalam satu sesi. Dikirim ulang ke LLM tiap turn. Terbatas oleh context window, jadi sering perlu diringkas kalau panjang.

Long-term memory

Pengetahuan yang bertahan lintas sesi: profil klien, dokumen, preferensi. Disimpan di database/vector store, lalu diambil saat relevan (retrieval).

memory.jsjs
// Short-term: simpan turn, ringkas kalau kepanjangan
function buildContext(history, maxTurns) {
  if (history.length <= maxTurns) return history;
  const recent = history.slice(-maxTurns);
  const summary = "[ringkasan " + (history.length - maxTurns) + " turn awal]";
  return [summary, ...recent];
}

Step 07 / Project

Project: Research Agent

Input topik -> riset web -> laporan otomatis

Sekarang gabungkan semuanya. Research Agent menerima sebuah topik, merencanakan pencarian, memakai tool web search, menyimpan temuan di memory, lalu menyusun laporan terstruktur. Ini project resmi F-04 Sesi 2.

Arsitektur (5 konsep digabung)

  • Planning: pecah topik jadi 2-3 sub-pertanyaan.
  • Tool calling: web_search untuk tiap sub-pertanyaan.
  • Memory: kumpulkan hasil di array temuan.
  • HITL (opsional): minta approval sebelum publish.
  • Reflection: cek apakah laporan punya cukup sumber.
