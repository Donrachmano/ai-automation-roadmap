◈ Step 2 / 8 — Sub-workflows & Modularity

Satu Workflow Raksasa = Satu Bencana Menunggu

Bayangkan CRM automation-mu tumbuh jadi 60 node dalam satu canvas. Setiap perubahan kecil berisiko merusak bagian lain, debugging jadi mimpi buruk, dan tidak ada bagian yang bisa dipakai ulang. Solusinya sama seperti function di JavaScript: pecah jadi unit kecil yang punya satu tanggung jawab.

// DUA NODE KUNCI

  • Execute Sub-workflow (di workflow induk/parent) — memanggil workflow lain, mengirim data, dan menunggu hasilnya kembali.
  • Execute Sub-workflow Trigger (di workflow anak/child) — pintu masuk sub-workflow; menerima data dari parent.
Analogi JS — sub-workflow itu function

  


  
// KAPAN HARUS DIPECAH?

  

    
SinyalAksi

    
Blok node yang sama muncul di 2+ workflowEkstrak jadi sub-workflow (DRY principle)

    
Workflow > 20–30 node dan sulit dibacaPecah per tanggung jawab: intake, enrich, notify

    
Ada logic yang ingin dites terpisahSub-workflow bisa dijalankan & di-debug mandiri

    
Tim/klien beda butuh versi berbedaParent beda, child (logic inti) tetap satu

  


  
// PASSING DATA PARENT ⇄ CHILD

  
Data mengalir seperti argumen dan return value sebuah function:

  
    
    
  • Parent mengirim items ke child lewat node Execute Sub-workflow.
    • 
    
  • Child menerima lewat Execute Sub-workflow Trigger — pilih mode input: Accept all data atau definisikan field spesifik agar kontraknya jelas.
    • 
    
  • Output node terakhir di child dikembalikan ke parent sebagai hasil.
    • 
  


  

    
◈ N8N CONTEXT — POLA NYATA

    
Sub-workflow "Format & Validate Lead": menerima raw lead → validasi email → normalisasi nama → return lead bersih. Dipanggil oleh workflow Form Intake, workflow CSV Import, dan workflow API Webhook. Satu logic, tiga pintu masuk. Ubah validasinya sekali, semua ikut terupdate.

  


  

    
! JEBAKAN UMUM

    
Sub-workflow yang dipanggil parent tidak otomatis muncul di execution list dengan jelas — aktifkan opsi "Save execution data" di setting sub-workflow agar tetap bisa di-debug. Dan hindari rantai terlalu dalam (parent → child → grandchild → ...) karena debugging jadi berlapis.

  


  







  
◈ Step 3 / 8 — Loop & Batching Data

  
n8n Sudah Looping — Kamu Hanya Perlu Mengendalikannya


  
Fakta yang sering bikin kaget: kebanyakan node n8n otomatis berjalan sekali per item. Kirim 50 items ke node Gmail, n8n mengirim 50 email tanpa kamu menulis loop apapun. Jadi kapan butuh loop manual?


  
    
    
  • Saat API tujuan punya rate limit — kamu harus mengirim per batch + jeda.
    • 
    
  • Saat memanggil API dengan pagination — ambil halaman demi halaman sampai habis.
    • 
    
  • Saat satu request harus berisi sekelompok item (misal: bulk insert 100 baris sekali kirim).
    • 
  


  
// NODE LOOP OVER ITEMS (SPLIT IN BATCHES)

  
Node ini memecah items menjadi batch berukuran N, lalu punya dua output:

  
    
    
  • loop — batch saat ini, diproses oleh node-node berikutnya, lalu dihubungkan balik ke node Loop untuk batch selanjutnya.
    • 
    
  • done — jalan setelah semua batch selesai (untuk rekap/notifikasi akhir).
    • 
  


  

    
◈ N8N CONTEXT — POLA RATE LIMIT

    
Loop Over Items (batch size 10) → HTTP Request → Wait node 2 detik → kembali ke Loop. Hasil: 10 request per ~2 detik, aman dari limit API. Ini pola standar untuk sinkronisasi CRM ↔ mailing list.

◈ N8N CONTEXT — TERNARY ADALAH SENJATA UTAMA

    
Di Sesi 2 F-02 kamu belajar ternary. Di expressions, ternary menggantikan node IF untuk kasus sederhana: {{ $json.score > 80 ? 'follow-up hari ini' : 'nurture email' }} — langsung di field, tanpa node tambahan.

  


  

    
! JEBAKAN UMUM

    
Expression hanya boleh satu ekspresi, bukan beberapa statement. Tidak bisa menulis if blok atau const x = ... di dalamnya — kalau butuh multi-langkah, itu tugasnya Code Node. Gunakan juga ?. agar field yang kadang kosong tidak meledakkan workflow.

  


  







  
◈ Step 5 / 8 — Custom Variables & Environments

  
Hardcode Hari Ini, Menderita Besok


  
Workflow-mu menyebut URL spreadsheet di 7 node berbeda. Klien minta ganti spreadsheet. Kamu edit 7 tempat, lupa 1, dan data bocor ke sheet lama selama seminggu. Inilah kenapa konfigurasi harus terpusat.


  
// TIGA LAPIS KONFIGURASI

  

    
LapisAksesUntuk apa

    
Custom Variables$vars.namaVarNilai bisnis yang dipakai lintas workflow: ID spreadsheet, nama channel Slack, ambang skor lead

    
Environment Variables$env.NAMA_VARKonfigurasi level instance (self-hosted): base URL, mode, timezone

    
Credentialsdipasang di nodeRahasia: API key, token, password — JANGAN pernah taruh di $vars/expression

  


  

    
Sebelum vs sesudah — refactor ke $vars

    

  


  
// DEV vs PRODUCTION ENVIRONMENT

  
Prinsip profesional: jangan pernah bereksperimen di workflow yang sedang dipakai klien. Pola pemisahannya:

  
    
    
  • Duplikat workflow — versi "(DEV)" non-aktif untuk eksperimen; versi production tetap jalan.
    • 
    
  • Variables per environment — $vars.sheetId menunjuk sheet test di dev, sheet asli di production. Workflow-nya identik, datanya beda.
    • 
    
  • Self-hosted: instance terpisah — n8n dev di lokal/Docker, n8n production di Railway/Render. $env membedakan keduanya.
    • 
    
  • n8n Environments (fitur Enterprise/Cloud plan tertentu) — sinkronisasi dev → production via Git. Cukup tahu konsepnya untuk sekarang.
    •

◈ Step 6 / 8 — Production Deployment & Monitoring

  
Workflow Production: Gagal Itu Pasti, Diam-Diam Gagal Itu Pilihan


  
API down, kuota Google habis, format data berubah — kegagalan pasti terjadi. Yang membedakan automation engineer profesional: sistemnya tahu dirinya gagal, memberitahu manusia, dan menyimpan jejak untuk investigasi.


  
// LAPIS 1 — ERROR HANDLING DI LEVEL NODE

  
    
    
  • Retry On Fail — node setting; coba ulang otomatis (mis. 3x, jeda 5 detik). Obat ampuh untuk error sementara (timeout, 429, 503).
    • 
    
  • On Error → Continue — item gagal tidak menghentikan seluruh eksekusi; bisa dialirkan ke jalur error terpisah (Continue using error output).
    • 
    
  • Timeout — batasi berapa lama node boleh menunggu, agar workflow tidak menggantung selamanya.
    • 
  


  
// LAPIS 2 — ERROR WORKFLOW (JARING PENGAMAN GLOBAL)

  
Buat satu workflow khusus berisi node Error Trigger. Lalu di setting workflow utama, pilih workflow itu sebagai Error Workflow. Setiap kali workflow utama gagal, error workflow otomatis jalan — membawa data: workflow mana, node mana, pesan error apa, jam berapa.


  

    
Blueprint error workflow standar

    

  


  
// LAPIS 3 — LOGGING

  
Execution log bawaan n8n bagus untuk debug, tapi untuk klien kamu butuh log yang bisa dibaca non-teknis: append satu baris ke Google Sheets "Log" setiap run — timestamp, jumlah item diproses, status, pesan error. Skill Sheets dari Sesi 2 langsung terpakai.


  

    
n8n Code Node — menyusun log entry

    

  


  
// DEPLOYMENT & MONITORING RUTIN

  
    
    
  • n8n Cloud — aktifkan workflow = deployed. Pastikan timezone instance benar agar Schedule Trigger akurat.
    • 
    
  • Self-hosted (Railway/Render + Docker) — kendali penuh & murah; konsekuensinya kamu yang jaga uptime, update versi, dan backup database workflow.
    • 
    
  • Heartbeat check — workflow kecil terjadwal yang ping sistem penting tiap jam; kalau gagal → alert. Kamu tahu sistem mati sebelum klien tahu.
    • 
    
  • Audit mingguan — buka daftar executions, filter status Error, cari pola: node mana paling sering gagal?
    •

◈ Step 8 / 8 — Debrief & Checkpoint

  
Status: Advanced n8n — Unlocked


  

    
SUB-WORKFLOWS
Execute Sub-workflow + trigger-nya = function untuk workflow. Satu logic, banyak pemanggil. Pecah saat >20-30 node atau ada duplikasi.

    
LOOP & BATCHING
Node berjalan per item secara default. Loop Over Items + Wait = senjata melawan rate limit. Jangan lupa sambungkan balik ke node Loop.

    
EXPRESSIONS
{{ }} = JS satu baris di field node. $json, $('Node'), $now, $vars. Ternary & ?. menggantikan banyak node IF sederhana.

    
VARIABLES & ENV
$vars untuk konfigurasi bisnis, $env untuk level instance, Credentials untuk rahasia. Dev dan production tidak pernah bercampur.

    
PRODUCTION
Retry On Fail → error output → Error Workflow global → log sheet → heartbeat. Sistem yang tahu dirinya gagal = sistem yang dibayar mahal.

    
PROJECT
CRM automation lengkap dengan error handling & logging — arsitektur dirakit di n8n Cloud, otak Code Node lolos validator.
