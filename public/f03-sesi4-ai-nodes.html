OpenAI & Anthropic Nodes

Ini node LLM paling dasar: kamu kirim prompt, node memanggil API (OpenAI atau Anthropic), dan jawaban model kembali sebagai item JSON. Konsepnya persis F-01: request → response. Bedanya, n8n yang mengurus HTTP call, auth header, dan parsing-nya untukmu.

Setup Credentials

  • Buat API key di platform masing-masing (platform.openai.com / console.anthropic.com)

STEP 3 / 8 · MATERI 2

AI Agent Node: Planning & Execution

Node LLM biasa = satu kali tanya, satu kali jawab. AI Agent berbeda: dia diberi tujuan dan tools, lalu dia sendiri yang merencanakan langkah dan mengeksekusinya — bisa beberapa putaran — sampai tujuan tercapai.

Siklus Kerja Agent: Plan → Act → Observe

1. PLAN
model menalar: "tool apa yang kubutuhkan?"
→
2. ACT
panggil tool (HTTP, Sheets, dll)
→
3. OBSERVE
baca hasil tool
→
ULANG / SELESAI
jawab final ke user

Contoh nyata: user bertanya "status order #1234 apa?". Agent menalar bahwa dia butuh data order → memanggil tool pencarian order → membaca hasilnya → baru menyusun jawaban. Semua itu terjadi di dalam satu node.

Anatomi AI Agent di n8n

AI Agent adalah node cluster — dia butuh sub-node yang ditempelkan ke port bawahnya:

SUB-NODEPERANWAJIB?
Chat ModelOtaknya — OpenAI Chat Model atau Anthropic Chat ModelYa
MemoryIngatan percakapan (Step 4)Opsional
ToolsTangan-tangannya: HTTP Request Tool, Code Tool, Call n8n Workflow Tool, Google Sheets Tool, dllOpsional tapi inilah inti agent
◈ N8N CONTEXT — TOOL ADALAH WORKFLOW SKILL LAMA-MU
Perhatikan daftar tools: HTTP Request Tool (= F-01), Code Tool (= F-02), Call n8n Workflow Tool (= sub-workflow dari Sesi 3). Agent tidak menggantikan skill lamamu — dia mengonsumsi semuanya. Sub-workflow yang sudah kamu modularisasi di Sesi 3 bisa langsung dijadikan tool untuk agent.

System Message: Pagar untuk Agent

Karena agent mengambil keputusan sendiri, system message jadi alat kontrol utamamu. Pola yang baik:

// Field "System Message" pada AI Agent
Kamu agent customer support Toko ABC.

ATURAN:
- Selalu jawab dalam Bahasa Indonesia.
- Untuk pertanyaan status order, WAJIB pakai tool "cari_order".
- Jangan pernah mengarang nomor resi.
- Jika data tidak ditemukan, katakan jujur dan tawarkan eskalasi.
⚠ JEBAKAN UMUM
Memberi agent terlalu banyak tools tanpa instruksi kapan memakainya → agent jadi lambat, mahal, dan sering salah pilih tool. Mulai dari 1–2 tools dengan deskripsi tool yang sangat jelas, baru tambah bertahap. Deskripsi tool itu dibaca oleh model — tulis seperti dokumentasi API mini (skill F-01 Sesi 3!).

STEP 4 / 8 · MATERI 3

Memory & Context Management

Fakta penting: LLM itu stateless. Setiap pemanggilan API adalah lembar kosong — model tidak ingat percakapan sebelumnya. Kalau user bilang "ya, yang itu" di pesan kedua, tanpa memory agent tidak tahu "itu" merujuk ke apa.

Simple Memory (Window Buffer)

Sub-node memory paling umum di n8n. Cara kerjanya: menyimpan N pesan terakhir per sesi, lalu menyuntikkannya kembali ke prompt setiap eksekusi.

PARAMETERFUNGSI
Session KeyIdentitas percakapan — pembeda antara user A dan user B
Context Window LengthBerapa pesan terakhir yang diingat (mis. 10)
◈ N8N CONTEXT — SESSION KEY DARI DATA NYATA
Session key diisi dengan expression dari trigger. Contoh: 
// Bot Telegram (integrasi Sesi 2):
{{ $json.message.chat.id }}

// Webhook form dengan email:
{{ $json.body.email }}
Dengan ini, setiap pelanggan punya "ingatan" terpisah — chat Budi tidak bocor ke chat Sari.

Context Window = Anggaran Token

Semua yang diingat (system prompt + riwayat + dokumen + pertanyaan baru) harus muat dalam context window model — dan semuanya dihitung sebagai token berbayar. Konsekuensinya:

  • Window length besar = konteks kaya, tapi biaya naik di setiap pesan
  • Window length kecil = murah, tapi agent cepat "lupa" awal percakapan
  • Praktik umum CS bot: 5–10 pesan terakhir sudah cukup
⚠ KAPAN TIDAK PERLU MEMORY
Workflow satu-arah (klasifikasi email, summarize dokumen, generate laporan harian) tidak butuh memory — tidak ada percakapan lanjutan. Menambahkan memory di sana hanya menambah biaya dan kompleksitas. Memory hanya untuk interaksi multi-giliran (chatbot, agent percakapan).

Latihan: Simulasi Window Buffer

Pahami mekanismenya dengan menulis logikanya sendiri — ambil N pesan terakhir dari riwayat.

Document Processing (PDF, Word)

Banyak data bisnis hidup di dalam file: invoice PDF, kontrak Word, CV pelamar. LLM tidak bisa membaca file biner langsung — alurnya selalu: ekstrak teks dulu, baru kirim ke model.

Binary Data di n8n

File di n8n mengalir sebagai binary data — menempel pada item, terpisah dari JSON-nya. Node Extract from File membaca binary itu dan menghasilkan field teks di JSON output:

// Sesudah Extract from File (operation: PDF / DOCX)
{
  "text": "INVOICE No. 2026-0042\nPT Maju Jaya\nTotal: Rp 12.500.000 ...",
  "numberOfPages": 2
}
JENIS FILECARA EKSTRAK
PDF (teks)Extract from File → operation Extract From PDF
Word (.docx)Extract from File → operation Extract From DOCX (hasil: teks polos)
PDF hasil scan (gambar)Teks tidak terbaca extractor biasa — butuh model vision / OCR sebagai jalur khusus

Pola Prompt untuk Dokumen

Setelah teks ada di $json.text, pola prompt yang efektif untuk ekstraksi data terstruktur:

Berikut isi sebuah invoice:
---
{{ $json.text }}
---
Ekstrak ke JSON dengan key persis berikut:
nomor_invoice, nama_vendor, total, tanggal_jatuh_tempo.
Jawab HANYA dengan JSON valid, tanpa teks lain.

Output JSON-nya lalu di-parse dan dialirkan ke Google Sheets / Airtable — pipeline F-01 + F-02 + F-03 bertemu di satu titik.

⚠ DOKUMEN BESAR = MASALAH TOKEN
Dokumen 50 halaman bisa melebihi context window atau membakar biaya. Strategi bertahap:
1. Trim — kalau yang dibutuhkan hanya halaman pertama (mis. invoice), potong teksnya di Code Node: text.slice(0, 8000).
2. Chunking — pecah dokumen jadi beberapa bagian, proses per bagian (pakai loop/batching dari Sesi 3), gabungkan hasilnya.
◈ N8N CONTEXT — DARI MANA FILE DATANG?
Sumber binary paling umum, semuanya sudah kamu kenal: Gmail Trigger (download attachments = ON), Webhook/Form dengan file upload, HTTP Request yang mengunduh file dari URL, dan Google Drive node.

STEP 6 / 8 · MATERI 5

Multi-step AI Workflow

Satu prompt raksasa yang disuruh "klasifikasi, lalu jawab, lalu format, lalu nilai sendiri" hasilnya rapuh. Sistem production memecahnya jadi beberapa langkah AI kecil yang masing-masing punya satu tugas — persis prinsip modularity dari Sesi 3, diterapkan ke AI.

Pola Inti: Classify → Route → Specialize → Validate

AI #1: Classify
temp 0.2 — label intent
→
Switch
route per kategori
→
AI #2: Generate
prompt khusus kategori
→
Code: Validate
cek output sebelum kirim
  • Classify — AI pertama hanya memberi label: "billing" / "teknis" / "umum". Tugas sempit = akurat.
  • Route — node Switch (Sesi 1!) mengarahkan item berdasarkan label
  • Specialize — tiap cabang punya AI dengan system prompt khusus domainnya
  • Validate — Code Node memeriksa output AI sebelum menyentuh pelanggan / database

◈ N8N CONTEXT — STRUCTURED OUTPUT PARSER
Untuk node berbasis chain/agent, n8n menyediakan sub-node Structured Output Parser: kamu definisikan skema JSON yang diharapkan, dan n8n memaksa + memvalidasi output model agar sesuai skema. Ini versi "built-in" dari Code Node validator di atas — pakai yang mana pun, prinsipnya sama: jangan percaya output AI mentah-mentah.
⚠ ERROR HANDLING UNTUK AI NODES
Tiga lapis pertahanan untuk workflow AI production (semua teknik dari Sesi 3):
1. Node setting Retry On Fail — API LLM kadang timeout / rate limit (ingat F-01 Sesi 3)
2. Fallback branch — output invalid → template jawaban aman / eskalasi manusia
3. Error Workflow + logging — semua kegagalan tercatat, seperti CRM-mu di Sesi 3

STEP 7 / 8 · PROJECT PRAKTIK

AI-Powered Customer Support Workflow

Project resmi roadmap Minggu 9–12. Kamu akan merakitnya di n8n Cloud — di sini kita kunci dulu blueprint-nya, lalu validator akan memeriksa kelengkapan desainmu.

◈ MISI
Bangun workflow customer support otomatis dengan spesifikasi minimal:
  • Trigger — pesan pelanggan masuk (Webhook / Telegram / Gmail)
  • Memory — Simple Memory dengan session key unik per pelanggan
  • Klasifikasi intent — AI pertama melabel: billing / teknis / umum (temp rendah)
  • AI Agent + tool — minimal 1 tool nyata (cek order via HTTP/Sheets, atau FAQ lookup)
  • Document processing — jika pelanggan melampirkan PDF/Word → Extract from File → masuk konteks AI
  • Validasi + fallback — output dicek; gagal → eskalasi ke manusia (Slack/email)
  • Balasan + logging — kirim jawaban ke pelanggan, catat semua tiket ke Google Sheets

Tulis Blueprint-mu (JSON)

Tulis rencana workflow-mu sebagai JSON: array nodes berisi langkah berurutan, masing-masing dengan type dan tujuan. Template awal sudah disediakan — lengkapi bagian yang masih "...", lalu klik VALIDASI.

BLUEPRINT · JSON
Hasil validasi akan muncul di sini...
  • ○Trigger penerima pesan (webhook / telegram / gmail)
  • ○Memory dengan session key dinamis (bukan nilai statis)
  • ○Langkah klasifikasi intent (classify / intent)
  • ○AI Agent dengan minimal 1 tool
  • ○Document processing (extract pdf / docx)
  • ○Fallback / eskalasi saat AI gagal
  • ○Balasan ke pelanggan + logging ke Sheets
◈ SETELAH BLUEPRINT LOLOS
